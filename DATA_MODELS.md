# Data Models

## Overview
This document describes the data structures used throughout the system.

The system involves three main data representations:
1. Motion event generated by ESP32
2. Motion event stored in MongoDB
3. Motion event used by the dashboard

---

## 1. ESP32 Motion Event (Sensor Output)

This is the data format created on the ESP32 after a motion event ends.

```json
{
  "sensor_id": "zone_1",
  "zone_name": "entry",
  "duration_ms": 27700,
  "duration_seconds": 27.7,
  "device_status": "online"
}
```

### Field Descriptions

| Field | Type | Description |
|------|------|-------------|
| sensor_id | string | Unique identifier for the sensor |
| zone_name | string | Physical location of the sensor |
| duration_ms | number | Duration of motion in milliseconds |
| duration_seconds | number | Duration of motion in seconds |
| device_status | string | Status of the device |

---

## 2. MongoDB Stored Motion Event

After the backend receives the event, it adds a timestamp and stores it in MongoDB.

```json
{
  "sensor_id": "zone_1",
  "zone_name": "entry",
  "duration_ms": 27700,
  "duration_seconds": 27.7,
  "device_status": "online",
  "received_at_utc": "2026-04-23T18:32:10Z"
}
```

### Additional Field

| Field | Type | Description |
|------|------|-------------|
| received_at_utc | string | UTC timestamp added by backend |

---

## 3. Dashboard Data Model

The dashboard fetches motion events from the backend and processes them in JavaScript.

### Used for:
- Total event count
- Most active zone
- Total duration per zone
- Movement path visualization
- Filtered event table

---

## Derived Data (Computed in Dashboard)

These values are not stored directly in the database but are calculated dynamically:

### Zone Statistics
```text
zoneStats = {
  entry: {
    count: number,
    total_duration: number
  },
  hallway: {...},
  room: {...}
}
```

### Movement Path Data
- Extracted from recent events
- Ordered by timestamp
- Consecutive duplicate zones removed

Example:
```
Entry → Hallway → Room
```

---

## Notes

- All timestamps are stored in **UTC format**
- Data is stored in:
  - Database: `smart_room`
  - Collection: `motion_events`
- Data is inserted as JSON documents using PyMongo
- No data normalization is applied (simple document model)

---

## Design Considerations

- Event-based model reduces unnecessary data storage
- JSON format simplifies communication between ESP32 and backend
- MongoDB allows flexible schema for future expansion
- Derived analytics handled on frontend for simplicity
